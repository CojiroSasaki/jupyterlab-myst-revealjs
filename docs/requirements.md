# jupyterlab-myst-revealjs: 要求仕様・設計哲学（統合版）

---

## 1. プロジェクトの位置づけ

**本プロジェクト（jupyterlab-myst-revealjs）は、
JupyterLab 上で MyST Markdown ノートブックを reveal.js スライドとして
ライブ表示する拡張である。**

Jupyter Book エコシステムにおいては、Web ページ・PDF に加えて
**スライド**を3つ目の出力形態として提供する。
ただし、Jupyter Book のインストールを前提とせず、
JupyterLab + jupyterlab-myst の環境であれば単体で動作することを目指す。
Sphinx 系をバックエンドにした文書作成ワークフローにおいても利用可能にするためである。

とはいえ、本プロジェクトの振る舞いの基準は Jupyter Book である。
Jupyter Book が定義するタグの解釈・表示方法を、スライドモードにおいても
可能な限り踏襲する。


## 2. 背景と動機

### 2.1 解決すべき問題

Jupyter Book 2 + MyST Markdown は研究ドキュメンテーション（実験ノート、論文執筆）
において優れた環境を提供しているが、
**研究発表や大学講義で必要なプレゼンテーション機能が欠けている**。

既存のスライドソリューションは大きく2つのアプローチに分かれるが、
いずれも Jupyter Book エコシステムの要件を満たしていない。

#### (A) HTML ジェネレーター型（ソース → ビルド → スライド HTML）

Markdown 等のソースからスライド HTML を静的に生成するアプローチ。

| ツール | 特徴 | 問題 |
|---|---|---|
| **Quarto + reveal.js** | Pandoc ベース。Markdown → reveal.js HTML。機能豊富 | MyST Markdown ではなく独自 Markdown 方言。ライブ実行不可 |
| **sphinx-revealjs** | Sphinx ベース。reST/MyST → reveal.js HTML | ビルドが必要。ライブ実行不可 |
| **Marp** | Markdown → スライド。シンプル | MyST 非対応。ライブ実行不可 |
| **nbconvert --to slides** | Notebook → reveal.js HTML | 静的出力のみ。MyST 非対応。ライブ実行不可 |

これらに共通する根本的な制約は **ライブ実行ができない** ことである。
生成されたスライド HTML にはカーネルが存在せず、
プレゼンテーション中にコードを実行してその場で結果を見せることができない。
研究発表でのインタラクティブなデモや、講義中に聴衆の質問に
コードで答えるといった体験は、この方式では実現不可能である。

#### (B) ライブ Notebook 型（Notebook = スライド、カーネル接続あり）

Notebook をそのままスライドとして表示し、カーネルとの接続を維持するアプローチ。

| ツール | 特徴 | 問題 |
|---|---|---|
| **RISE** | Jupyter Notebook 用 reveal.js 拡張。ライブ実行可能 | classic notebook 専用（≤6）。開発終了。MyST 非対応 |
| **jupyterlab-rise** | RISE の JupyterLab 移植。ライブ実行可能 | 約2年間更新なし。JLab 4.x で 404 エラー。MyST 非対応。図の centering/resize 不可 |

これらはライブ実行が可能であり、本プロジェクトの思想的な先行者である。
しかし、MyST Markdown への対応に消極的であり、
基本的なレイアウト機能も欠如している。開発が停滞しており、
将来にわたって改善される見込みは低い。

#### 本プロジェクトが埋めるギャップ

本プロジェクトは (B) のライブ実行を継承しつつ、
(A) が持つ表現力（reveal.js の最新機能、リッチなマークアップ）を
MyST Markdown のネイティブサポートによって実現する。

**本プロジェクトの焦点はスライドショー時のライブ実行である。**
これは静的 HTML 生成では絶対に代替できない機能であり、
Jupyter Book エコシステムに欠けている唯一のピースである。

### 2.2 なぜ MyST Markdown が重要か

- LaTeX/Typst ほどの複雑さなしに美しい文書が書ける
- 記述量が少なく、ドキュメント作成コストが低い
- プレゼンテーションの表現力に十分な機能を持つ
  （図のサイズ・配置制御、数式、admonition 等）
- 旧 RISE が外部拡張（jupyter_contrib_nbextensions）に頼っていた機能の多くを
  Jupyter Book のタグ体系で解決できる（§4.4 参照）


## 3. 設計哲学

### 3.1 「Notebook がスライドになる」思想

**最も重要な設計原則**: 本プロジェクトは「MyST Markdown からスライドをコンパイルする」
ツールではない。**Notebook そのものがライブスライドになる**。

§2.1 で述べた通り、既存ソリューションには HTML ジェネレーター型 (A) と
ライブ Notebook 型 (B) がある。本プロジェクトは明確に (B) に属する。

- Quarto / sphinx-revealjs / Marp 的アプローチ
  （ソース → ビルド → スライド HTML）**ではない**
- RISE 的アプローチ
  （Notebook = スライド、カーネルは生きている）**を継承する**

(A) の HTML ジェネレーター型がどれほど美しいスライドを生成しても、
プレゼンテーション中のライブ実行は不可能である。
この一点において、(A) と (B) の間には越えられない断絶がある。
本プロジェクトの存在意義はこの断絶の (B) 側にある。

### 3.2 ビューレイヤーとしてのスライドショー

**本プロジェクトは JupyterLab ノートブックのビューレイヤーである。**

JupyterLab は同じファイルに対して複数の独立したビューを開く仕組み
（DocumentRegistry）を持っている。本プロジェクトはこの仕組みを利用し、
ノートブックとは別の独立したパネルとしてスライドビューを提供する。
ノートブックの DOM に手を加えることはない。

コンテンツの管理・カーネル接続・セル実行はすべて JupyterLab に委ね、
本プロジェクトはノートブックのモデル（セル一覧、メタデータ、出力）を
読み取ってスライドとして表示する。

スライドビューはノートブックのモデルを参照しているため、
セルの実行結果が更新されれば、スライドのビューもその変更を反映する。

この原則により:
- カーネル管理の再実装は不要（ノートブックと同じカーネルを共有する）
- セル実行の仕組みを独自に持つ必要がない
- hide-input 等の表示制御はビューの問題であり、実行の可否とは独立する
- ユーザーはスライドモード中にトグルで入力を展開し、
  編集・再実行・折りたたみが可能

### 3.3 リアルタイム実行

プレゼンテーション中にコードセルをリアルタイムに実行できることが本質的な価値:

- 講義でのライブデモ
- 計算結果をその場で確認
- 聴衆の質問にその場でコードを書いて答える
- 研究発表でのインタラクティブな可視化

これは静的スライドでは絶対に得られない体験であり、本プロジェクトの存在意義の核心。
§3.2 のビューレイヤー原則により、この機能は JupyterLab のカーネル接続を
そのまま利用する形で自然に実現される。

### 3.4 ユーザー移行の互換性

既存の RISE / jupyterlab-rise ユーザーがそのまま移行できることを重視:

- **セルメタデータ互換**: `slideshow.slide_type` をそのまま踏襲
- **操作互換**: Ctrl+R（Mac: Option+R）でスライドモード切替
- **Property Inspector** でスライドタイプ選択

ただし、内部アーキテクチャの互換性は不要。

### 3.5 プレゼンテーションの方向性

- **発表者 → 聴衆の一方向**を基本とする
- 双方向機能（multiplex 等）は当面スコープ外
- JupyterBook エコシステムで双方向を実現する意義が現時点では不明確


## 4. 設計原則

### 4.1 4つの原則

**原則 0（最上位）: シンプルな実装を保つために、機能を制限する**

機能を実現するために複雑な実装をするのではなく、
シンプルな実装を維持できる範囲で機能を提供する。
これは本プロジェクトの最上位の設計原則であり、
以下の原則 1〜3 はこの原則に奉仕するために存在する。

- コンテンツにレイアウト指示を書かない（原則 1）→ 本プロジェクトが解釈すべき構文を最小化
- Jupyter Book のタグ体系に準拠する（原則 2）→ 独自仕様の設計・実装・文書化を回避
- スライド固有の制御を限定する（原則 3）→ 本プロジェクトの責務範囲を明確に制限

将来的に高機能を実現する際には複雑化が避けられない場面もあり得る。
そのためにも、機能要件を事前にしっかり調査し、
後から大きな設計変更が発生しないようにすることが重要である。

この原則は以下の判断基準を提供する:
- ある機能を実装する際、シンプルな方法で実現できるか？
- できない場合、その機能は本当に必要か？ 後回しにできないか？
- 複雑な実装を避けるために、機能の仕様を限定できないか？
- reveal.js が既に提供している機能を再実装していないか？

**原則 1: コンテンツにはレイアウト指示を書かない**

MyST ディレクティブは出力形態に依存しない意味的構造を記述するために使う。
スライド固有のレイアウト（横並び等）を `{grid}` で記述すると、
PDF 出力で不都合が生じる。

**原則 2: セルタグの解釈は Jupyter Book に準拠する**

hide-input, remove-cell 等のタグは Jupyter Book（myst-nb）が定義する体系に従う。
本プロジェクトは独自のタグを発明しない。
Jupyter Book のドキュメントを参照すれば、スライドモードでの挙動も予測できる。

**原則 3: スライド固有の制御は slideshow メタデータに限定する**

スライド境界、背景画像・色、スライド固有レイアウト等は
slideshow メタデータで指定する。
これらは Jupyter Book のビルドでは無視されるため、
PDF/Web 出力に影響を与えない。

### 4.2 動機

同一の Notebook が複数の出力形態を持つ:

- **スライド**: 横長（16:9 等）、1スライド = 限られた面積
- **Web ページ**: Jupyter Book による HTML 出力、縦スクロール
- **PDF 文書**: 縦長（A4 等）、ページ分割

横長のスライドでは、説明文と図を横に並べないと収まらない場面が多い。
しかし、同じ内容を縦長の PDF にするとき、横並びは不要どころか有害
（カラムが狭くなる、図がはみ出す）。

### 4.3 3つのレイヤー

#### レイヤー 1: コンテンツ（MyST ディレクティブ）

すべての出力形態で共通に解釈される意味的構造。

| 要素 | 役割 | 例 |
|---|---|---|
| `{figure}` | 「これは図である」 | width, alt, caption |
| `{code-cell}` | 「これは実行可能コードである」 | 言語指定 |
| `{math}` | 「これは数式である」 | LaTeX 記法 |
| `{admonition}` | 「これは注意書きである」 | note, warning, tip |
| 見出し、段落、リスト | テキスト構造 | Markdown 記法 |

これらは出力形態に依存しない。スライドでも PDF でも同じ意味を持つ。

`{grid}` や `{tab-set}` はレイアウト指示を含む構文であり、
出力形態によって望ましい振る舞いが異なる。
ユーザーが意図的に使うことは妨げないが、
スライド固有のレイアウトにこれらを使うことは推奨しない。

#### レイヤー 2: 表示制御（セルタグ / セルメタデータ）

JupyterLab のセルには2種類の付加情報がある:
- **セルタグ**: 文字列のリスト（例: `["hide-input", "gridwidth-1-2"]`）。
  JupyterLab の Property Inspector で追加・削除できる。
- **セルメタデータ**: JSON の構造化データ（例: `{"slideshow": {"slide_type": "slide"}}`）。

本プロジェクトは両方を使用する。以下にそれぞれの用途を整理する。

**Jupyter Book 互換タグ（全出力形態で有効）:**

Jupyter Book が定義するタグの解釈を、スライドモードでも踏襲する。

| セルタグ | Jupyter Book での挙動 | スライドでの挙動 |
|---|---|---|
| `hide-input` | トグルボタン付き折りたたみ（入力非表示、クリックで展開） | **同左を踏襲** |
| `hide-output` | トグルボタン付き折りたたみ（出力非表示、クリックで展開） | **同左を踏襲** |
| `hide-cell` | トグルボタン付き折りたたみ（セル全体非表示） | **同左を踏襲** |
| `remove-input` | 入力を完全除去（トグルなし） | **同左を踏襲** |
| `remove-output` | 出力を完全除去（トグルなし） | **同左を踏襲** |
| `remove-cell` | セルを完全除去（トグルなし） | **同左を踏襲** |

**スライド固有の表示制御（スライドのみ有効、Jupyter Book では無視）:**

| 種類 | キー / タグ名 | 用途 |
|---|---|---|
| セルメタデータ | `slideshow.slide_type` | スライド境界定義（slide, sub-slide, fragment, skip, notes） |
| セルタグ | `gridwidth-1-2`, `gridwidth-1-3` 等 | セル幅制御（横並び表示） |
| セルメタデータ（要設計） | 未定 | 背景画像・背景色 |

#### レイヤー 3: 出力形態固有の表現

各パイプラインが最終出力を生成する際の固有処理。

| 出力形態 | 処理 | 担当 |
|---|---|---|
| スライド | セルを reveal.js の section 構造に配置 | **本プロジェクト** |
| スライド | reveal.js テーマ・遷移・chalkboard 適用 | **本プロジェクト** |
| スライド | Jupyter Book 互換のタグ解釈・表示 | **本プロジェクト** |
| Web ページ | MyST → HTML ビルド | Jupyter Book |
| PDF | MyST → LaTeX/PDF ビルド | Jupyter Book |

### 4.4 旧 RISE 外部拡張の代替

旧 RISE が `jupyter_contrib_nbextensions` に頼っていた機能と、
本プロジェクトでの解決方法:

| 旧 RISE の問題 | 外部拡張での対処 | 本プロジェクトでの解決 |
|---|---|---|
| 図の centering / resize | 不可能（RISE 単体） | `{figure}` の width, align（レイヤー1: MyST 構文） |
| コード入力の非表示 | Hide Input 拡張（cell.metadata.hide_input） | `hide-input` タグ（レイヤー2: Jupyter Book 準拠） |
| コード出力の非表示 | なし | `hide-output` タグ（レイヤー2: Jupyter Book 準拠） |
| 2カラムレイアウト | Split Cell 拡張 | `gridwidth-*` タグ（レイヤー2: スライド固有） |
| 背景画像 | CSS ハック | slideshow メタデータ（レイヤー2: スライド固有） |
| スライド装飾 | CSS ハック | admonition, card 等 MyST 構文 + reveal.js テーマ |
| 黒板機能 | RISE 組み込み | reveal.js chalkboard プラグイン |


## 5. 本プロジェクトの責務の境界

### 5.1 本プロジェクトが行うこと

本プロジェクトの独自実装範囲は以下の3つに限定される:

1. **セルメタデータ / タグ → reveal.js スライド構造へのマッピング**
   - `slideshow.slide_type` を読み、セルを reveal.js の section に配置する
   - セルの中身のレンダリングは JupyterLab の RenderMimeRegistry
     （セル内容をレンダリングする共通基盤）を経由して
     jupyterlab-myst 等の既存レンダラーに委譲する
2. **ノートブック状態とスライドビューの同期**
   - ノートブックのモデルを監視し、セル実行結果の更新をスライドに反映する
   - ノートブックとスライドビューは同じ DocumentContext を共有するため、
     カーネル接続は自動的に共有される
3. **Jupyter Book 互換タグのスライドモード内での表示制御**
   - hide-input 等のタグをスライドモード内で解釈し、
     Jupyter Book と同等の表示（トグルボタン付き折りたたみ等）を提供する
   - gridwidth-* タグを CSS クラスに変換してセル幅を制御する

reveal.js の機能（テーマ、遷移、chalkboard 等）は reveal.js に委ね、
本プロジェクトは設定を渡す橋渡しを行う。

### 5.2 本プロジェクトが行わないこと

1. **コンテンツのレンダリング**: jupyterlab-myst に委ねる
2. **カーネル管理・セル実行**: JupyterLab に委ねる（ビューレイヤー原則）
3. **静的 HTML エクスポート**: Jupyter Book のビルドパイプラインに委ねる
4. **アクセシビリティの独自実装**: jupyterlab-myst（意味的 HTML）と
   reveal.js（ベースライン）の出力を壊さないことで対応
5. **JupyterLab 通常モードでの表示制御**: スコープ外
6. **独自タグの発明**: Jupyter Book 既存のタグ体系を尊重する
7. **MyST ディレクティブの独自実装**: 車輪の再発明をしない

  `gridwidth-1-2`, `gridwidth-1-3` 等のセルタグで CSS のみ制御

- スライドごとの個別 CSS スタイリング

#### テーマ / カスタマイズ
- デフォルトテーマの選定
- ユーザーによる CSS カスタマイズの提供方法
- ノートブックメタデータでのテーマ指定

#### 既存ノートブック形式以外の入力
- MyST Markdown ファイル（.myst.md）からの直接スライド表示
- Jupytext 形式のサポート

### 6.4 非機能要件

#### パフォーマンス
- スライドモード開始は即座（< 1秒）
- スライド遷移は滑らか（reveal.js ネイティブ性能を損なわない）

#### 互換性
- JupyterLab 4.x 以上
- jupyterlab-myst との共存（必須依存）
- 主要ブラウザ対応（Chrome, Firefox, Safari, Edge）

#### 保守性
- reveal.js のバージョンアップに追従しやすい設計
- JupyterLab のメジャーアップデートへの追従コストを最小化


## 7. 技術的前提

### 7.1 構築方針
- **スクラッチから開発**（jupyterlab-rise のフォークではない）
- JupyterLab 拡張（TypeScript）
- reveal.js 6.0（最新）をベースとする

### 7.2 フォークしない理由
- jupyterlab-rise の `/rise/` スタンドアロンアプリ方式が JLab 4.x で破綻
- 依存関係（TypeScript 5.0.4, 旧 reveal.js）が古い
- MyST 対応にはレンダリングパイプラインの根本的書き換えが必要
- 残すべきコードがほとんどない

### 7.3 再利用する要素
- **セルメタデータ形式**: `slideshow.slide_type` の仕様
- **UX パターン**: Ctrl+R トグル、Property Inspector でのスライドタイプ選択
- **jupyterlab-myst**: MyST レンダリングエンジン（必須依存）

### 7.4 JupyterLab との統合方式

**DocumentRegistry に新しい WidgetFactory を登録し、
.ipynb ファイルに対する独立したスライドビューを提供する。**

この方式は JupyterLab が元々持つ「同じファイルに対して
複数のビューを開く」仕組みに乗るものであり、
ノートブックの DOM を変形せず、別 URL のルーティングも不要。
詳細は design.md §1.2 を参照。

ユーザーの起動方法:
- Ctrl+R（Mac: Option+R）キーバインドでスライドビューを開く
  （内部的に WidgetFactory を通じて DocumentWidget を生成）
- 右クリックメニュー「Open With > Slideshow」でも開ける

### 7.5 依存関係とスタンドアローン動作

**ネットワーク接続なしで動作することを理想とする。**

講義室や学会会場のネットワーク環境は不安定であることが多い。
プレゼンテーションツールが外部 CDN に依存していると、
最も重要な場面で動作しないリスクがある。

したがって、reveal.js はバンドルに含める（CDN 読み込みではない）。

```
jupyterlab-myst-revealjs
├── 必須依存: jupyterlab-myst（コンテンツレンダリング）
├── 推奨依存: jupyterlab-gridwidth 等（スライド用レイアウト拡張）
└── バンドル:  reveal.js 6.0（パッケージに同梱）
```

**注**: Jupyter Book は必須依存ではない。
JupyterLab + jupyterlab-myst の環境があればスライドショーは動作する。
Jupyter Book は Web ページ / PDF ビルドを行う際にのみ必要。


## 8. 論点の整理状況

### 8.1 決定済み

#### プロジェクトの名称
**jupyterlab-myst-revealjs** に決定。

- 技術スタック（JupyterLab + MyST + reveal.js）が名前から読み取れる
- Jupyter Book に限定しない（JupyterLab 単体でも動作可能）
- RISE ブランドは継承しない
  （本プロジェクトは RISE のフォークではなく、新規プロジェクトであるため）

#### ライセンス
**MIT** とする。reveal.js (MIT) および jupyterlab-myst (MIT) と整合する。

#### コミュニティとの関係
- 最初は**独立リポジトリ**で個人ユースを想定する
- 完成度を見て **jupyterlab-contrib** への参加を検討する
- paugier/myst-pres はコンパイル型を志向しており、思想が異なる

#### reveal.js の組み込み方式
**バンドルに含める**（§7.5 参照）。
ネットワーク接続なしのスタンドアローン動作を理想とするため、
CDN 読み込みは採用しない。

### 8.2 MVP（Minimum Viable Product）

#### MVP の方針

**3つの責務すべてをカバーするが、各責務の中の機能を最小限にする。**

責務の一部を欠いた状態で開発を始めると、後から追加する際に
設計変更が発生するリスクがある（原則 0）。
したがって MVP は、アーキテクチャ全体を薄く通す「垂直スライス」とする。

加えて、**外部拡張のタグをスライドビューに反映できるか**を
MVP の段階で検証する。jupyterlab-gridwidth のタグ対応を含めることで、
将来的に他の拡張のタグも同じパターンで取り込めることを保証する。

このパターンは hide-input / remove-cell と同一であり、
確立されれば将来的に他の拡張にも同じ方法で対応可能。

#### MVP の完了基準

以下がすべて動作すること:

1. `slideshow.slide_type` が slide / skip のセルを持つ Notebook を開く
2. Ctrl+R でスライドモードに入る
3. MyST Markdown（figure, admonition, 数式等）がスライド内で正しく表示される
4. コードセルをスライドモード中に実行し、出力がスライドに反映される
5. hide-input タグ付きセルのコード入力がトグルで折りたたまれている
6. remove-cell タグ付きセルがスライドに表示されない
7. jupyterlab-gridwidth で幅指定（`gridwidth-1-2` タグ等）されたセルが、スライド内で指定幅で表示される
8. ブラウザ F11 でフルスクリーン表示ができる

### 8.3 将来的に検討

以下の論点は、基本機能の実装と安定化を優先し、後回しとする。

原則 0（シンプルさ）とビューレイヤー原則（§3.2）に基づき、
各論点の**実際の担当**を明確にする。
本プロジェクトが独自に実装すべきものは極めて限定的であり、
多くは既存エコシステムの機能に委ねられる。

#### reveal.js に委ねるもの

| 論点 | reveal.js の該当機能 | 本プロジェクトの作業 |
|---|---|---|
| **PDF エクスポート** | print-pdf モード | 設定を露出するだけ |
| **キーボードナビゲーション** | 組み込みナビゲーション | 設定不要（デフォルトで有効） |
| **高コントラストモード** | テーマ CSS カスタマイズ | 高コントラスト用テーマ CSS を提供（将来） |
| **スライド縦横比** | `width` / `height` オプション | メタデータから設定値を渡す |
| **サードパーティプラグイン** | reveal.js プラグイン API | API を露出する（将来） |

#### Jupyter Book / jupyterlab-myst に委ねるもの

| 論点 | 担当 | 本プロジェクトの作業 |
|---|---|---|
| **静的 HTML エクスポート** | Jupyter Book のビルドパイプライン | なし（スコープ外） |
| **スクリーンリーダー対応** | jupyterlab-myst（意味的 HTML 生成）+ reveal.js（ベースライン） | 既存の出力を壊さないこと |

#### 調査タスク（実装ではなく情報収集）

- **Jupyter Book のタグ解釈の正確な仕様**: myst-nb のソースコードから
  hide-input 等のトグル表示の正確な挙動、CSS 構造、
  カスタムプロンプト（code_prompt_show / code_prompt_hide）の仕様を確認する

#### 本プロジェクトの独自実装範囲

上記の分類から明らかなように、本プロジェクトが独自に実装する範囲は
§5.1 に記載した3項目に限定される。
これ以外はすべて reveal.js、jupyterlab-myst、Jupyter Book に委ねる。

## 付録: 設計判断の経緯

本文書は以下の議論を経て策定された。

1. **Jupyter Book エコシステムにおけるプレゼン機能の欠如**を問題として定義
2. **「Notebook がスライドになる」思想**を確認（Beamer / sphinx-revealjs 型ではない）
3. **セルメタデータ（slideshow.slide_type）の RISE 互換性**を確定
4. **jupyterlab-rise のフォーク vs スクラッチ**を技術的に検討し、スクラッチを選択
5. **コンテンツとプレゼンテーションの分離原則**を策定
   - `{grid}` 等のレイアウト構文をスライド用に使わない
   - メタデータ/タグで出力形態別の制御を行う
6. **jupyterlab-hidecell の調査**から、hide-input 等の表示制御は
   Jupyter Book の挙動を基準として本プロジェクトが実装すべきと判断
7. **「本プロジェクトは Jupyter Book のスライドショー出力」**という位置づけを確定
8. **ビューレイヤー原則**を確立
   - カーネル・セル実行は JupyterLab に委ねる
   - hide-input はビューの問題であり、実行とは独立
9. **既存ソリューションの全体像を整理**し、本プロジェクトの焦点を明確化
   - HTML ジェネレーター型（Quarto, sphinx-revealjs 等）vs ライブ Notebook 型（RISE 系）
   - 本プロジェクトの焦点はスライドショー時のライブ実行であり、
     静的 HTML 生成では代替不可能であることを強調
10. **シンプルさの原則と未整理論点の整理**
    - 名称を jupyterlab-myst-revealjs に決定（RISE ブランド不継承、JupyterBook 非限定）
    - ライセンス: MIT、独立リポジトリから開始
    - シンプルさを設計原則の最上位（原則 0）に位置づけ、他の原則がそれに奉仕する構造に
    - スタンドアローン動作（reveal.js バンドル同梱）を決定
    - 後回し論点を reveal.js 組み込み機能 vs 独自実装に分類
    - MVP は技術調査・アーキテクチャ設計後に決定
11. **MVP を定義**
    - 3つの責務すべてを薄く実装する「垂直スライス」方針
    - jupyterlab-gridwidth 対応を含め、外部拡張タグ反映パターンを早期検証
    - 完了基準を8項目で明示
